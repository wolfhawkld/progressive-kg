# 分布式训练

> 将数据、参数、激活或专家状态分配到多设备，并协调通信以突破容量或缩短训练时间。

## 先明确目标

分布式训练可能为了两件不同的事：

- **容量扩展**：单设备放不下模型、优化器状态或激活。
- **吞吐扩展**：模型能放下，但希望用更多设备缩短 wall time。

并行会引入通信、同步、气泡和故障恢复成本；小模型或慢网络上，多设备可能反而更慢。

## 并行维度

每种并行方式都在切分某类状态并交换另一类状态；选择时应先定位真正的容量或吞吐瓶颈。

### 复制式数据并行

经典 DDP 在每个 data-parallel rank 保存完整模型副本，处理不同 micro-batch，并在反向时 all-reduce 梯度。

- 适合单个设备能容纳完整训练状态、主要目标是提高吞吐的情况。
- 全局 batch 通常为 `micro_batch × grad_accum × data_parallel_world_size`。
- 梯度是 sum 还是 average、loss reduction 和学习率缩放必须核对。

经典 DDP 的模型和优化器状态通常仍需单卡容纳；梯度累积不会解决这部分容量问题。

### Sharded Data Parallel

ZeRO/FSDP 在 data-parallel ranks 之间分片状态：

| 级别 | 主要分片对象 |
|---|---|
| ZeRO-1 | 优化器状态 |
| ZeRO-2 | 再分片梯度 |
| ZeRO-3 / full shard | 再分片参数，并在计算前后 all-gather/reshard |

它降低每卡常驻内存，但增加通信、临时峰值和参数生命周期复杂度。FSDP 的具体 API 正在演进，应按项目锁定版本查官方文档。

### 张量、流水线与其他并行

| 方式 | 切分对象 | 主要代价 |
|---|---|---|
| Tensor Parallel | 单层矩阵维度 | 高频 collective 与布局约束 |
| Pipeline Parallel | 层/阶段 | micro-batch 气泡、调度和跨阶段激活通信 |
| Sequence/Context Parallel | 序列维度 | 注意力通信和 mask/位置处理 |
| Expert Parallel | MoE 专家/token | all-to-all 与负载不均 |

“3D 并行”通常指 data + tensor + pipeline，但真实系统还可叠加 sequence 和 expert parallel。

## 选择步骤

1. 在单设备上建立可复现、数值正确的基线。
2. 分解峰值内存：参数、梯度、优化器、激活、临时 workspace。
3. 若模型能放下，先尝试 DDP；若训练状态放不下，再评估 FSDP/ZeRO；单层矩阵仍放不下时考虑 tensor parallel。
4. 层数很深且设备组较大时，再评估 pipeline；MoE 按专家布局考虑 expert parallel。
5. 用 profiler 量化计算、collective、数据加载和 idle time，不按“模型参数量”表格机械选方案。

## 正确性检查

- 每个 rank 的初始参数、数据 shard 和随机种子符合设计。
- 梯度累积期间只在预定 step 同步，且尾批处理一致。
- BatchNorm、dropout、采样器和动态 loss scaling 的跨 rank 语义明确。
- checkpoint 包含模型、优化器、scheduler、scaler、采样位置和并行布局元数据。
- 先比较一个 optimizer step 后的参数，再比较短程 loss 曲线，最后才测吞吐。

## 性能指标

- **吞吐**：samples/tokens per second。
- **扩展效率**：$N$ 设备速度提升除以理想 $N$ 倍。
- **time-to-quality**：达到同一验证质量所需时间。
- **模型 FLOP 利用率**：需要统一 FLOP 口径。
- **通信占比与峰值显存**：平均显存不足以发现 all-gather 峰值。

## 关系网络

- 组合 [[混合精度训练]] — 降低通信与部分张量内存，但数值状态需一致
- 同步 [[梯度下降优化]] — collective 和 reduction 改变全局梯度语义
- 应用 [[MoE架构]] — expert parallel 依赖 all-to-all 与容量均衡
- 影响 [[Muon优化器]] — 二维矩阵后处理需适配参数分片
- 诊断 [[训练调试]] — 分布式死锁、数据重复和恢复偏移需专门检查
